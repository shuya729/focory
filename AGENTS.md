# AGENTS.md

## 自動品質チェック（Ultracite / Biome）

このプロジェクトでは **Ultracite**（ゼロコンフィグの Biome プリセット）により、厳格なコード品質基準を Formatter / Lint で自動適用します。Ultracite は Biome（Rust 製の高速 Lint/Formatter）をベースにしており、多くの問題は自動修正可能です。

### クイックリファレンス

* コード整形・自動修正: `pnpm fix`
* 問題チェック: `pnpm check`

### 運用方針

* コミット前に `pnpm fix` を実行する
* 自動修正できる問題はツールに任せる
* 自動では判断できない内容（命名・設計・業務ロジック・UX）は人間が責任を持つ

## コア原則（全言語・全レイヤー共通）

### 1. 可読性優先

* コードは「書く時間」より「読まれる時間」のほうが長い
* 明確な変数名・関数名
* コメントより自己文書化コード
* フォーマットと記法の一貫性
* 短さより意図の明確さ

### 2. KISS

* 動作する最もシンプルな解を選ぶ
* 過剰設計・早すぎる最適化を避ける
* 巧妙さより理解しやすさ

### 3. DRY

* 重複ロジックは関数・モジュールに抽出
* 再利用可能なコンポーネント/ユーティリティ
* コピペ実装を避ける
  ※ ただし「無理な共通化」で可読性を落とさない

### 4. YAGNI

* 将来必要か不明な機能を先回り実装しない
* 推測による抽象化・一般化を避ける
* 必要になった時点で複雑性を導入
* まずはシンプルに作り、必要に応じてリファクタリング

### 5. 目標品質

コードは以下を満たすことを目指す。

* アクセシブル
* 高性能
* 型安全
* 保守しやすい
* 意図が明示的

## コーディングスタイル（共通）

## 不変性（重要）

既存データをその場で変更せず、**新しい値を作って返す**。

* 既存オブジェクト/配列の直接変更を避ける
* 更新時はコピーを作って変更を反映
* 状態更新では「前の値」に依存する安全な更新方法を使う

### 不変性を守る理由

* 隠れた副作用を防ぐ
* デバッグ容易
* 変更点の追跡が容易
* 並行/非同期処理で安全性が上がる

## TypeScript / JavaScript 標準

### 命名規則（変数）

* 1文字変数を避け、意味のある名前
* 真偽値は `is/has/can/should`
* 数値は意味や単位が分かるようにする

例（方針）:

* 良い: `marketSearchQuery`, `isUserAuthenticated`, `totalRevenue`
* 悪い: `q`, `flag`, `x`

### 命名規則（関数）

* 動詞 + 目的語（または述語）
* `is/has/should` は真偽値返却関数
* 名詞だけの曖昧な関数名を避ける

例（方針）:

* 良い: `fetchMarketData`, `calculateSimilarity`, `isValidEmail`
* 悪い: `market`, `similarity`, `email`

### 型安全性

* `any` は極力使わない
* 不明な型は `unknown`
* 必要箇所で type narrowing
* 明確さが増すなら引数/戻り値に明示的型
* 不変リテラルに `as const`
* 型アサーションより型絞り込みを優先

### モダン JavaScript / TypeScript の推奨

* コールバックや短い関数ではアロー関数
* `.forEach()` や添字ベース `for` より `for...of`
* 安全なアクセスに `?.`（optional chaining）
* デフォルト値に `??`（nullish coalescing）
* 文字列連結よりテンプレートリテラル
* オブジェクト/配列の分割代入
* デフォルトは `const`、再代入が必要な時だけ `let`、`var` は使わない

### Async / Promise ベストプラクティス

* async 関数内では Promise を必ず `await` し、戻り値の使い忘れをしない
* Promiseチェーンより `async/await`（可読性）
* 非同期処理は適切に `try-catch` でエラーハンドリング
* 並列化可能な独立処理はまとめて実行（逐次実行を避ける）
* `new Promise(async (...) => ...)` の async executor は使わない

### エラーハンドリング

* エラーは各レイヤーで明示的に扱う
* 失敗しうる処理にエラーハンドリングを付ける
* 文字列ではなく `Error` を投げる
* メッセージは具体的に（ただし機密情報は含めない）
* `try-catch` は「捕まえる意味がある場合」に使う（再throwだけなら不要）
* エラーを握りつぶさない

### デバッグ用コードの扱い

* 本番コードに `console.log` / `debugger` / `alert` を残さない
* 必要なログは構造化し、機密情報を除外して記録する

## React / JSX ベストプラクティス

### コンポーネント設計

* クラスより関数コンポーネント
* props は明示的に型付け
* コンポーネントを別コンポーネント内で定義しない
* `children` は props に直接渡すよりネストする
* `key` はユニークIDを優先し、配列 index は避ける（やむを得ない場合を除く）

### Hooks のルール

* Hooks はトップレベルのみ（条件分岐/ループ内で呼ばない）
* 依存配列は正しく（不足・過剰に注意）
* 再利用ロジックはカスタムフックへ

### 状態管理

* 前状態に依存する更新は関数型更新
* stale state を避ける
* 状態の責務を明確にし、過剰な分割/統合を避ける

### 条件付きレンダリング

* 条件ごとに読みやすく分ける
* ネストした三項演算子（ternary hell）を避ける
* エラー/ローディング/データ表示の分岐は明確にする

### JSX とアクセシビリティ（重要）

* 意味のある HTML 要素（`button`, `nav`, `main`, `label` など）
* フォーム入力にラベル
* 画像に適切な alt
* 見出し階層（h1-h6）を正しく
* マウス依存操作はキーボード操作も考慮
* ARIA は必要なときに正しく（過剰使用しない）

### React 19+ の指針

* `React.forwardRef` より、可能な場面では `ref` を props として扱う設計を優先

## API 設計標準

### REST API 規約（例）

* `GET /api/resources` : 一覧取得
* `GET /api/resources/:id` : 単一取得
* `POST /api/resources` : 作成
* `PUT /api/resources/:id` : 全体更新
* `PATCH /api/resources/:id` : 部分更新
* `DELETE /api/resources/:id` : 削除

### クエリパラメータ

* フィルタ・ページング・ソートはクエリで表現
* 例: `status`, `limit`, `offset`, `page`, `sort`, `order`
* API全体で命名と意味を統一する

### レスポンス形式の一貫性

成功/失敗を判別しやすい共通形にする（推奨要素）:

* `data`（成功時）
* `error`（失敗時）
* `meta`（ページング等）

`meta` 例:

* `total`, `page`, `limit`

### 入力検証（API境界で必須）

* すべてのユーザー入力を処理前に検証
* スキーマベース検証を優先
* 失敗時は早期に返す（fail fast）
* エラーメッセージは明確にする
* 外部データ（APIレスポンス、ファイル内容、ユーザー入力）を信用しない

## ファイル構成とコード構成

## ファイル構成の原則

* 少数の巨大ファイルより多数の小さなファイル
* 高凝集・低結合
* 型別ではなく機能/ドメイン単位
* 大きくなったモジュールからユーティリティや責務を抽出

### 目安

* 通常: 200〜400行
* 上限目安: 800行
* 関数: 50行未満（必要なら分割）

## 典型的なディレクトリ構造（例）

* `src/app`（画面/ルーティング）
* `src/components`（UI・フォーム・レイアウト）
* `src/hooks`（カスタムフック）
* `src/lib`（ユーティリティ・設定・APIクライアント等）
* `src/types`（型定義）
* `src/styles`（スタイル）

※ 配置は採用技術に合わせるが、責務分離の考え方は共通。

## ファイル命名の原則

* コンポーネント: `PascalCase`
* フック: `use` + `camelCase`
* ユーティリティ: `camelCase`
* 型定義ファイル: 規約を決めて統一（例: `.types` サフィックス）
* 名前から責務が分かるようにする

## コード組織化の原則

* 関数は単一責務で認知負荷を低く
* 複雑な条件式は意味ある真偽値変数へ抽出
* 早期 return でネストを減らす
* ネスト三項より単純な条件分岐
* 関連コードを近くに置き、関心事を分離
* バレルファイル（何でも再エクスポートする `index`）は避ける

## コメントとドキュメント

### コメントを書く基準

コメントは WHAT ではなく **WHY** を説明する。

良い対象:

* 例外的な実装判断
* パフォーマンス上の意図
* 回避策・制約の背景
* 一見不自然に見えるコードの理由

避ける対象:

* コードを読めば分かる説明
* 自明な処理（代入・インクリメント等）

### パブリック API / 共有関数のドキュメント

必要に応じて:

* 何をする関数か
* 引数（意味・制約・デフォルト値）
* 戻り値
* 例外/エラー条件
* 利用例（必要な場合）

### 自己文書化コードを優先

* 長いコメントより適切な命名・分割
* 複雑ロジックのみ補助的にコメント

---

## パフォーマンス ベストプラクティス

### 一般原則

* 早すぎる最適化は避ける
* ただし明らかな無駄（不要な逐次処理・過剰取得・不要再計算）は早めに除去
* 計測できるものは計測する

### UI/React系

* 高コスト計算はメモ化を検討
* コールバック再生成が問題ならメモ化を検討
* 重いコンポーネントは遅延読み込みを検討
* 画像はフレームワークの最適化コンポーネントを優先（例: Next.js の画像コンポーネント）

### データ取得・DBクエリ

* 必要な列だけ取得（`SELECT *` を避ける）
* ページング/制限件数を適切に
* 高コスト検索・集計はキャッシュやレート制御を併用
* クエリ責務を明確にし、不要な結合や変換を避ける

### 実装レベルの注意（Ultracite統合）

* ループ内の累積処理で毎回 spread による新配列/新オブジェクト生成をしない（性能悪化）
* ループ内で正規表現を生成せずトップレベルで再利用
* 名前空間 import より必要なものを明示的に import

## テスト標準

### テスト構造（AAA）

* Arrange（準備）
* Act（実行）
* Assert（検証）

### テスト命名

* 期待される振る舞いが分かる
* 入力条件と結果が読み取れる
* `works` のような曖昧な名前は避ける

### テスト実装の原則

* アサーションは `test()` / `it()` 内に書く
* 非同期は `done` ではなく `async/await`
* `.only` / `.skip` をコミットしない
* `describe` の過剰ネストを避け、フラットに保つ

### セキュリティ観点の自動テスト（推奨）

最低限テストする:

* 認証が必要なAPIで未認証アクセスが拒否される
* 認可（ロール/権限）がない操作が拒否される
* 不正入力がバリデーションで拒否される
* レート制限が発動する

## コードスメル（アンチパターン）検出

### 1. 長い関数

* 目安: 50行超は分割を検討
* 入力検証 / 変換 / 永続化 / レスポンス生成を1関数に詰め込まない
* 小さな関数へ分け、意図が読める構造にする

### 2. 深いネスト

* 4レベル以下が目安（5+ は要改善）
* 早期 return / ガード節で浅くする
* 異常系を先に返し、正常系を下に残す

### 3. マジックナンバー

* 意味不明な数値を直接書かない
* 意味ある定数名に抽出
* 単位（ms, bytes, retries など）が分かる命名

## セキュリティガイドライン

セキュリティはオプションではありません。1つの脆弱性がサービス全体を危険にさらす可能性があります。迷った場合は「厳しめ」に判断してください。

### 必須セキュリティチェック（コミット前）

すべてのコミット前に確認:

* [ ] ハードコードされたシークレットなし（APIキー、パスワード、トークン）
* [ ] すべてのユーザー入力が検証済み
* [ ] SQLインジェクション防止（パラメータ化クエリ）
* [ ] XSS防止（ユーザーHTMLのサニタイズ、危険な描画回避）
* [ ] CSRF保護が有効
* [ ] 認証/認可が検証済み
* [ ] すべてのエンドポイントにレート制限
* [ ] エラーメッセージが機密データを漏らさない

## 1. シークレット管理

### 絶対にしないこと

* APIキー、トークン、パスワードをソースコードにハードコード
* テスト用でも本物の資格情報をコードに埋め込む
* シークレットをログやエラーに出力する

### 常にすること

* 環境変数またはシークレットマネージャーを使う
* 起動時に必要なシークレットの存在を検証する
* 本番シークレットはホスティング基盤のシークレット管理機能で設定する
* 露出した可能性があるシークレットはローテーションする
* `.env.local` などのローカルシークレットファイルを `.gitignore` に含める
* git履歴にシークレットが残っていないか確認する

### 検証ステップ

* [ ] ハードコードされたAPIキー/トークン/パスワードなし
* [ ] すべてのシークレットが環境変数またはシークレット管理にある
* [ ] ローカル環境ファイルが追跡対象外
* [ ] git履歴にシークレットが残っていない
* [ ] 本番シークレットはホスティング基盤側に設定済み

---

## 2. 入力検証（全境界で必須）

### 原則

* すべてのユーザー入力を処理前に検証
* スキーマベースの検証を優先
* ホワイトリスト方式で制約（ブラックリスト頼みを避ける）
* エラーは早期に返す
* エラーメッセージに機密情報を含めない
* 外部データ（APIレスポンス、ユーザー入力、ファイル内容）を信用しない

### ファイルアップロード検証（必須）

* サイズ制限（例: 最大5MBなど）
* MIMEタイプ制限
* 拡張子制限
* 必要に応じて中身の検査（Content-Type 偽装対策）
* 保存先の権限・公開範囲を明確化

### 検証ステップ

* [ ] 全ユーザー入力をスキーマで検証
* [ ] ファイルアップロードはサイズ・タイプ・拡張子を制限
* [ ] ユーザー入力をクエリ等へ直接流し込まない
* [ ] ホワイトリスト検証を採用
* [ ] エラー文言が機密情報を漏らさない

---

## 3. SQLインジェクション防止

### 禁止事項

* SQL文字列連結でユーザー入力を埋め込む
* 生SQLを組み立てる際にエスケープ頼みで対応する

### 推奨事項

* 常にパラメータ化クエリ
* ORM / クエリビルダーの安全な API を正しく使う
* 生SQLでもプレースホルダを使う
* フィルタ条件・ソートキー等も検証して扱う

### 検証ステップ

* [ ] すべてのDBクエリがパラメータ化されている
* [ ] SQL文字列連結がない
* [ ] ORM/クエリビルダーの安全なAPIを使用
* [ ] ユーザー入力を含む条件が検証済み

---

## 4. 認証と認可

### トークン保管

* トークンは `httpOnly` Cookie を優先
* `localStorage` 保存は XSS リスクの観点で避ける
* `Secure` / `SameSite` 属性を適切に設定する

### 認可（Authorization）

* 機密操作の前に認可チェック
* 「認証済み」だけでなくロール/権限を確認
* 管理者操作・他人データ操作はサーバー側で必ず検証

### セッション管理

* 有効期限を適切に
* 無効化（ログアウト・失効）を設計
* セッション固定攻撃などを考慮

### 検証ステップ

* [ ] トークンは httpOnly Cookie（localStorage回避）
* [ ] 機密操作前に認可チェック
* [ ] ロールベースアクセス制御を実装
* [ ] セッション管理が安全

---

## 5. XSS 防止

### 原則

* ユーザー提供HTMLをそのまま描画しない
* 必要な場合のみサニタイズして描画
* フレームワークのデフォルトエスケープを活用
* `dangerouslySetInnerHTML` など危険APIは極力避ける

### CSP（Content Security Policy）

* 本番では CSP ヘッダーを設定
* `script-src`, `style-src`, `img-src`, `connect-src` 等を明示
* 必要最小限にする（広すぎる許可を避ける）

### 検証ステップ

* [ ] ユーザー提供HTMLをサニタイズ
* [ ] CSPヘッダーを設定
* [ ] 未検証の動的コンテンツ描画なし
* [ ] フレームワークのXSS保護機能を活用

---

## 6. CSRF 保護

### 原則

* 状態変更（POST/PUT/PATCH/DELETE）にCSRF対策
* CSRFトークン検証
* Cookie は `SameSite=Strict` を基本方針として検討
* 必要に応じてダブルサブミットCookieパターン

### 検証ステップ

* [ ] 状態変更操作でCSRFトークンを検証
* [ ] すべてのCookieに適切な `SameSite` 属性
* [ ] ダブルサブミットCookie等（必要時）

---

## 7. レート制限

### 原則

* すべての API エンドポイントに適用
* 高コスト操作（検索、認証試行、重い集計、外部API呼び出し）はより厳しく
* IPベース +（可能なら）認証済みユーザーIDベース
* エラーメッセージは簡潔にし、内部情報を出しすぎない

### 検証ステップ

* [ ] 全APIエンドポイントでレート制限
* [ ] 高コスト操作に厳しい制限
* [ ] IPベース制限
* [ ] ユーザーベース制限（認証済み）

---

## 8. 機密データの露出防止（ログ・エラー）

### ロギング

* パスワード、トークン、APIキー、秘密情報をログに出さない
* 支払い情報・個人情報はマスキング/編集
* 監査ログとアプリログの目的を分ける

### エラーメッセージ

* ユーザー向けは一般的な文言
* 詳細（stack trace、内部構造）はサーバーログのみ
* 例外原文をレスポンスに含めない

### 検証ステップ

* [ ] ログにパスワード・トークン・シークレットなし
* [ ] ユーザー向けエラーは一般化
* [ ] 詳細エラーはサーバーログのみ
* [ ] スタックトレースをユーザーに露出しない

## 9. ブロックチェーンセキュリティ（Solana 等を扱う場合）

※ ブロックチェーン機能がないプロジェクトでは対象外。該当する場合は以下を必須とする。

### ウォレット署名検証

### トランザクション検証

* 宛先アドレス検証
* 金額/上限検証
* 残高確認
* ブラインド署名を避ける（内容表示を徹底）

### 検証ステップ

* [ ] ウォレット署名を検証
* [ ] トランザクション詳細を検証
* [ ] トランザクション前の残高確認
* [ ] ブラインド署名を避ける

## 10. 依存関係セキュリティ

### 原則

* 依存関係は定期的に更新
* 脆弱性スキャンを継続的に実施
* ロックファイルを必ずコミット
* CI/CD では再現可能インストール（例: `npm ci` 相当）
* Dependabot 等の自動更新支援を有効化

### 定期確認の観点

* audit による既知脆弱性
* 古いパッケージの放置
* 破壊的変更の影響評価
* メンテされていない依存の置換検討

### 検証ステップ

* [ ] 依存関係が最新に近い
* [ ] 既知の脆弱性が解消
* [ ] ロックファイルをコミット
* [ ] Dependabot等を有効化
* [ ] 定期セキュリティ更新を実施

## セキュリティテスト（自動化推奨）

以下の観点を自動テストに含める:

* 認証必須エンドポイントに未認証でアクセス→拒否
* 管理者権限必須エンドポイントに一般ユーザー→拒否
* 不正入力（形式/型/制約違反）が 400 系→拒否
* レート制限超過で 429
* CSRFトークン不正→拒否（該当構成）
* 認可ポリシーにより他人データへアクセス不可（該当構成）

## フレームワーク別の補足指針
