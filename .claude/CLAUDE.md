# CLAUDE.md

## 自動品質チェック（Ultracite / Biome）

このプロジェクトでは **Ultracite**（ゼロコンフィグの Biome プリセット）を用いて、厳格なコード品質基準を自動整形・Lint により適用します。
Ultracite は Biome（Rust製の高速なLint/Formatter）をベースにしており、多くの問題は自動修正可能です。

### クイックリファレンス

* コード整形・自動修正: `pnpm dlx ultracite fix`
* 問題チェック: `pnpm dlx ultracite check`
* 設定診断: `pnpm dlx ultracite doctor`

### 運用方針

* コミット前に `ultracite fix` を実行する
* 自動修正できる問題はツールに任せる
* 自動では判断できない内容（命名・設計・業務ロジック・UX）は人間が責任を持つ

## コア原則（全言語・全レイヤー共通）

### 1. 可読性優先

* コードは「書く時間」より「読まれる時間」のほうが長い
* 明確な変数名・関数名を使う
* コメントより自己文書化コードを優先する
* フォーマットと記法は一貫させる
* 短さよりも意図の明確さを重視する

### 2. KISS（Keep It Simple, Stupid）

* 動作する最もシンプルな解を選ぶ
* 過剰設計を避ける
* 早すぎる最適化を避ける
* 巧妙さより理解しやすさを優先する

### 3. DRY（Don’t Repeat Yourself）

* 重複ロジックは関数・モジュールに抽出する
* 再利用可能なコンポーネント/ユーティリティを作る
* コピペ実装を避ける
* ただし「無理な共通化」による可読性低下には注意する

### 4. YAGNI（You Aren’t Gonna Need It）

* 将来必要になるか不明な機能を先回り実装しない
* 推測による抽象化・一般化を避ける
* 必要になった時点で複雑性を導入する
* まずはシンプルに作り、必要に応じてリファクタリングする

### 5. 目標品質

コードは以下を満たすことを目指す。

* アクセシブル
* 高性能
* 型安全
* 保守しやすい
* 意図が明示的

## コーディングスタイル（共通）

## 不変性（重要）

既存のデータをその場で変更せず、**新しい値を作って返す**ことを原則とします。

* 既存オブジェクト/配列の直接変更を避ける
* 更新時はコピーを作って変更を反映する
* 状態更新では「前の値」に依存する安全な更新方法を使う

### 不変性を守る理由

* 隠れた副作用を防げる
* デバッグが容易になる
* 変更点の追跡がしやすくなる
* 並行処理/非同期処理で安全性が上がる

## TypeScript / JavaScript 標準

### 命名規則（変数）

* 1文字変数を避け、意味のある名前を付ける
* 真偽値には `is/has/can/should` などを使う
* 数値は何を表す値か明確にする（単位があるなら名前に含める）

例（方針）:

* 良い: `marketSearchQuery`, `isUserAuthenticated`, `totalRevenue`
* 悪い: `q`, `flag`, `x`

### 命名規則（関数）

* 動詞 + 目的語（または述語）を基本にする
* `is/has/should` は真偽値返却関数に使う
* 役割が不明な名詞だけの関数名を避ける

例（方針）:

* 良い: `fetchMarketData`, `calculateSimilarity`, `isValidEmail`
* 悪い: `market`, `similarity`, `email`

### 型安全性

* `any` は極力使わない
* 本当に不明な型には `unknown` を使う
* 必要な箇所で型を絞り込む（type narrowing）
* 明確さが増す場合は、関数の引数・戻り値に明示的な型を付ける
* 不変リテラルには `as const` を活用する
* 型アサーションより型絞り込みを優先する

### モダン JavaScript / TypeScript の推奨

* コールバックや短い関数ではアロー関数を使う
* `.forEach()` や添字ベース `for` より `for...of` を優先する
* 安全なプロパティアクセスには `?.`（optional chaining）を使う
* デフォルト値には `??`（nullish coalescing）を使う
* 文字列連結よりテンプレートリテラルを優先する
* オブジェクト/配列の分割代入を使う
* デフォルトは `const`、再代入が必要な時だけ `let`、`var` は使わない

### Async / Promise ベストプラクティス

* async 関数内では Promise を必ず `await` し、戻り値を使い忘れない
* Promiseチェーンより `async/await` を優先する（可読性向上）
* 非同期処理では適切に `try-catch` でエラーハンドリングする
* 並列化可能な独立処理はまとめて実行する（逐次実行を避ける）
* `new Promise(async (...) => ...)` のような async executor は使わない

### エラーハンドリング

* エラーは各レイヤーで明示的に扱う
* 失敗しうる処理にエラーハンドリングを付ける
* 文字列ではなく `Error` オブジェクトを投げる
* エラーメッセージは具体的にする（ただし機密情報は含めない）
* `try-catch` は「捕まえる意味がある場合」に使う（ただ再throwするだけなら不要）
* エラーを黙って握りつぶさない

### デバッグ用コードの扱い

* 本番コードに `console.log` / `debugger` / `alert` を残さない
* 必要なログは構造化し、機密情報を除外して記録する

## React / JSX ベストプラクティス

### コンポーネント設計

* クラスコンポーネントより関数コンポーネントを使う
* コンポーネントの props は明示的に型付けする
* コンポーネントを別コンポーネント内で定義しない
* `children` は props に直接渡すより、開始タグと終了タグの間にネストする
* `key` はユニークIDを優先し、配列 index の使用は避ける（やむを得ない場合を除く）

### Hooks のルール

* Hooks はトップレベルでのみ呼ぶ（条件分岐/ループ内で呼ばない）
* 依存配列は正しく記述する（不足・過剰に注意）
* 再利用ロジックはカスタムフックに抽出する

### 状態管理

* 前の状態に依存する更新は関数型更新を使う
* stale state（古い値参照）を避ける
* 状態の責務を明確にし、必要以上に分割・統合しない

### 条件付きレンダリング

* 条件ごとに読みやすく分ける
* ネストした三項演算子（ternary hell）を避ける
* エラー・ローディング・データ表示の分岐は明確にする

### JSX とアクセシビリティ（重要）

* 意味のある HTML 要素を使う（`button`, `nav`, `main`, `label` など）
* フォーム入力にはラベルを付ける
* 画像には適切な代替テキスト（alt）を付ける
* 見出し階層（h1-h6）を正しく使う
* マウスイベントに依存する操作にはキーボード操作も考慮する
* ARIA 属性は必要なときに正しく使う（過剰使用しない）

### React 19+ の指針

* `React.forwardRef` より、可能な場面では `ref` を props として扱う設計を優先する

## API 設計標準

### REST API 規約

基本方針（例）:

* `GET /api/resources` : 一覧取得
* `GET /api/resources/:id` : 単一取得
* `POST /api/resources` : 作成
* `PUT /api/resources/:id` : 全体更新
* `PATCH /api/resources/:id` : 部分更新
* `DELETE /api/resources/:id` : 削除

### クエリパラメータ

* フィルタ・ページング・ソートはクエリパラメータで表現する
* 例: `status`, `limit`, `offset`, `page`, `sort`, `order`
* API全体で命名と意味を統一する

### レスポンス形式の一貫性

レスポンスは成功/失敗を判別しやすい共通形にする。

推奨要素:

* `data`（成功時）
* `error`（失敗時）
* `meta`（ページングなどの付加情報）

`meta` の例:

* `total`
* `page`
* `limit`

### 入力検証（API境界で必須）

* すべてのユーザー入力を処理前に検証する
* スキーマベースの検証を優先する
* 失敗時は早期に返す（fail fast）
* エラーメッセージは明確にする
* 外部データ（APIレスポンス、ファイル内容、ユーザー入力）を信用しない

## ファイル構成とコード構成

## ファイル構成の原則

* 少数の巨大ファイルより、多数の小さなファイルを優先する
* 高凝集・低結合を目指す
* 型別ではなく「機能/ドメイン単位」で整理する
* 大きくなったモジュールからユーティリティや責務を抽出する

### 目安

* 通常: 200〜400行程度
* 上限目安: 800行
* 関数: 50行未満を目安（必要なら分割）

## 典型的なディレクトリ構造（例）

* `src/app`（画面/ルーティング）
* `src/components`（UI・フォーム・レイアウト）
* `src/hooks`（カスタムフック）
* `src/lib`（ユーティリティ・設定・APIクライアント等）
* `src/types`（型定義）
* `src/styles`（スタイル）

※ フレームワーク依存の配置は採用技術に合わせるが、責務分離の考え方は共通。

## ファイル命名の原則

* コンポーネント: `PascalCase`
* フック: `use` プレフィックス + `camelCase`
* ユーティリティ: `camelCase`
* 型定義ファイル: 規約を決めて統一（例: `.types` サフィックス）
* 名前から責務が分かるようにする

## コード組織化の原則

* 関数は単一責務で、認知負荷を低く保つ
* 複雑な条件式は意味のある真偽値変数に抽出する
* 早期 return を使ってネストを減らす
* ネストした三項演算子より単純な条件分岐を使う
* 関連するコードを近くに配置し、関心事を分離する
* バレルファイル（何でも再エクスポートする `index`）は避ける

## コメントとドキュメント

### コメントを書く基準

コメントは「何をしているか（WHAT）」ではなく、**なぜそうしているか（WHY）** を説明する。

良いコメントの対象:

* 例外的な実装判断
* パフォーマンス上の意図
* 回避策・制約の背景
* 一見不自然に見えるコードの理由

避けるべきコメント:

* コードを読めば分かる内容の説明
* 変数代入・インクリメントなど自明な説明

### パブリック API / 共有関数のドキュメント

公開関数・共有ユーティリティには、必要に応じて以下を記述する。

* 何をする関数か
* 引数（意味・制約・デフォルト値）
* 戻り値
* 例外/エラー条件
* 利用例（必要な場合）

### 自己文書化コードを優先

* 長いコメントより、適切な命名・分割で読めるコードを優先する
* 複雑なロジックだけを補助的にコメントする

---

## パフォーマンス ベストプラクティス

### 一般原則

* 早すぎる最適化は避ける
* ただし明らかな無駄（不要な逐次処理・過剰な取得・不要な再計算）は早めに除去する
* 計測できるものは計測する

### UI/React系

* 高コスト計算はメモ化を検討する
* コールバックの再生成が問題になる箇所ではメモ化を検討する
* 重いコンポーネントは遅延読み込みを検討する
* 画像はフレームワークが提供する最適化コンポーネントを優先する（例: Next.js の画像コンポーネント）

### データ取得・DBクエリ

* 必要な列だけ取得する（`SELECT *` を避ける）
* ページング/制限件数を適切に設定する
* 高コスト検索・集計にはキャッシュやレート制御を組み合わせる
* クエリの責務を明確にし、不要な結合や変換を避ける

### 実装レベルの注意（Ultracite統合）

* ループ内の累積処理で毎回 spread を使って新配列/新オブジェクトを作らない（パフォーマンス悪化）
* ループ内で正規表現を生成せず、トップレベルで再利用する
* 名前空間importより、必要なものを明示的に import する

## テスト標準

### テスト構造（AAA パターン）

各テストは以下の流れを明確にする。

* Arrange（準備）
* Act（実行）
* Assert（検証）

### テスト命名

* 期待される振る舞いが分かる名前にする
* 入力条件と結果が読み取れる命名にする
* `works` のような曖昧な名前は避ける

### テスト実装の原則

* アサーションは `test()` / `it()` ブロック内に書く
* 非同期テストで `done` コールバックを使わず、`async/await` を使う
* `.only` / `.skip` をコミットしない
* `describe` の過剰なネストを避け、構造をフラットに保つ

### セキュリティ観点の自動テスト（推奨）

最低限、以下をテストする。

* 認証が必要なAPIで未認証アクセスが拒否される
* 認可（ロール/権限）がない操作が拒否される
* 不正入力がバリデーションで拒否される
* レート制限が発動する

## コードスメル（アンチパターン）検出

### 1. 長い関数

* 目安: 50行超の関数は分割を検討
* 入力検証 / 変換 / 永続化 / レスポンス生成を1関数に詰め込まない
* 小さな関数へ分け、意図が読める構造にする

### 2. 深いネスト

* 4レベル以下を目安にする（5+ は要改善）
* 早期 return / ガード節を使ってネストを浅くする
* 異常系を先に返し、正常系を下に残す

### 3. マジックナンバー

* 意味不明な数値を直接書かない
* 意味のある定数名に抽出する
* 単位（ms, bytes, retries など）が分かる命名にする

## 1. シークレット管理

### 絶対にしないこと

* APIキー、トークン、パスワードをソースコードにハードコード
* テスト用でも本物の資格情報をコードに埋め込む
* シークレットをログやエラーに出力する

### 常にすること

* 環境変数またはシークレットマネージャーを使う
* 起動時に必要なシークレットの存在を検証する
* 本番シークレットはホスティング基盤のシークレット管理機能で設定する
* 露出した可能性があるシークレットはローテーションする
* `.env.local` などのローカルシークレットファイルを `.gitignore` に含める
* git履歴にシークレットが残っていないか確認する

### 検証ステップ

* [ ] ハードコードされたAPIキー/トークン/パスワードなし
* [ ] すべてのシークレットが環境変数またはシークレット管理にある
* [ ] ローカル環境ファイルが追跡対象外
* [ ] git履歴にシークレットが残っていない
* [ ] 本番シークレットはホスティング基盤側に設定済み

---

## 2. 入力検証（全境界で必須）

### 原則

* すべてのユーザー入力を処理前に検証
* スキーマベースの検証を優先
* ホワイトリスト方式で制約する（ブラックリスト頼みを避ける）
* エラーは早期に返す
* エラーメッセージに機密情報を含めない
* 外部データ（APIレスポンス、ユーザー入力、ファイル内容）を信用しない

### ファイルアップロード検証（必須）

* サイズ制限（例: 最大5MBなど）
* MIMEタイプ制限
* 拡張子制限
* 必要に応じて中身の検査（Content-Typeの偽装対策）
* 保存先の権限・公開範囲を明確化

### 検証ステップ

* [ ] 全ユーザー入力をスキーマで検証
* [ ] ファイルアップロードはサイズ・タイプ・拡張子を制限
* [ ] ユーザー入力をクエリ等へ直接流し込まない
* [ ] ホワイトリスト検証を採用
* [ ] エラー文言が機密情報を漏らさない

## 3. SQLインジェクション防止

### 禁止事項

* SQL文字列の連結でユーザー入力を埋め込む
* 生SQLを組み立てる際にエスケープ頼みで対応する

### 推奨事項

* 常にパラメータ化クエリを使う
* ORM / クエリビルダーを正しく使う
* 生SQLが必要でもプレースホルダを使う
* フィルタ条件・ソートキーなども検証した上で扱う

### 検証ステップ

* [ ] すべてのDBクエリがパラメータ化されている
* [ ] SQL文字列連結がない
* [ ] ORM/クエリビルダーの安全なAPIを使用
* [ ] ユーザー入力を含む条件が検証済み

---

## 4. 認証と認可

## トークン保管

* トークンは `httpOnly` Cookie を優先する
* `localStorage` への保存は XSS リスクの観点で避ける
* `Secure` / `SameSite` 属性を適切に設定する

## 認可（Authorization）

* 機密操作の前に認可チェックを行う
* 「認証済み」だけでなく「何ができるか（ロール/権限）」を確認する
* 管理者操作・他人データ操作は必ずサーバー側で検証する

## セッション管理

* セッションの有効期限を適切に設定する
* 無効化（ログアウト・失効）を設計する
* セッション固定攻撃などの対策を考慮する

## 行レベルセキュリティ（RLS）を使う場合

* テーブル単位でRLSを有効化
* SELECT / UPDATE / DELETE の各操作にポリシーを定義
* 「アプリ側の認可チェック」と「DB側ポリシー」の二重防御を意識する

### 検証ステップ

* [ ] トークンは httpOnly Cookie に保存（localStorage回避）
* [ ] 機密操作前に認可チェック
* [ ] ロールベースアクセス制御を実装
* [ ] セッション管理が安全
* [ ] （DB機能対応時）RLS が有効でポリシーが妥当

---

## 5. XSS 防止

### 原則

* ユーザー提供のHTMLをそのまま描画しない
* 必要な場合のみサニタイズして描画する
* フレームワークのデフォルトのエスケープ機構を活用する
* `dangerouslySetInnerHTML` のような危険なAPIは極力避ける

### CSP（Content Security Policy）

* 本番ではCSPヘッダーを設定する
* `script-src`, `style-src`, `img-src`, `connect-src` などを明示する
* 運用上必要な範囲に最小化する（広すぎる許可を避ける）

### 検証ステップ

* [ ] ユーザー提供HTMLをサニタイズ
* [ ] CSPヘッダーを設定
* [ ] 未検証の動的コンテンツ描画なし
* [ ] フレームワークのXSS保護機能を活用

---

## 6. CSRF 保護

### 原則

* 状態変更系操作（POST/PUT/PATCH/DELETE）にCSRF対策を入れる
* CSRFトークン検証を行う
* Cookieには `SameSite=Strict` を基本方針として検討する
* 必要に応じてダブルサブミットCookieパターンを使う

### 検証ステップ

* [ ] 状態変更操作でCSRFトークンを検証
* [ ] すべてのCookieに適切な `SameSite` 属性
* [ ] ダブルサブミットCookie等の対策（必要時）

---

## 7. レート制限

### 原則

* すべてのAPIエンドポイントにレート制限を適用する
* 高コスト操作（検索、認証試行、重い集計、外部API呼び出し）はより厳しくする
* IPベースに加え、認証済みユーザーIDベースでも制限する
* エラーメッセージは簡潔にし、過度な内部情報を出さない

### 検証ステップ

* [ ] 全APIエンドポイントでレート制限
* [ ] 高コスト操作に厳しい制限
* [ ] IPベース制限
* [ ] ユーザーベース制限（認証済み）

---

## 8. 機密データの露出防止（ログ・エラー）

## ロギング

* パスワード、トークン、APIキー、秘密情報をログに出さない
* 支払い情報・個人情報はマスキング/編集して記録する
* 監査ログとアプリログの目的を分ける

## エラーメッセージ

* ユーザー向けには一般的なエラーメッセージを返す
* 詳細な例外情報（stack trace、内部構造）はサーバーログのみに残す
* 例外の原文をそのままレスポンスに含めない

### 検証ステップ

* [ ] ログにパスワード・トークン・シークレットなし
* [ ] ユーザー向けエラーは一般化されている
* [ ] 詳細エラーはサーバーログのみ
* [ ] スタックトレースをユーザーに露出しない

## 9. ブロックチェーンセキュリティ（Solana 等を扱う場合）

※ ブロックチェーン機能がないプロジェクトでは対象外。
該当する場合は以下を必須とする。

### ウォレット署名検証

* ウォレット所有者の署名を検証する
* 署名対象メッセージに nonce / timestamp / domain を含める
* リプレイ攻撃対策を行う

### トランザクション検証

* 宛先アドレスを検証する
* 金額/上限を検証する
* 残高を確認する
* ユーザーにブラインド署名をさせない（内容表示を徹底）

### 検証ステップ

* [ ] ウォレット署名を検証
* [ ] トランザクション詳細を検証
* [ ] トランザクション前の残高確認
* [ ] ブラインド署名を避ける

## 10. 依存関係セキュリティ

### 原則

* 依存関係は定期的に更新する
* 脆弱性スキャンを継続的に実施する
* ロックファイルを必ずコミットする
* CI/CDでは再現可能なインストール方法を使う（例: `npm ci` 相当）
* 自動更新支援（Dependabot等）を有効化する

### 定期確認の観点

* 既知の脆弱性がないか（audit）
* 古いパッケージが放置されていないか
* 破壊的変更の影響評価
* メンテされていない依存関係の置換検討

### 検証ステップ

* [ ] 依存関係が最新に近い状態
* [ ] 既知の脆弱性が解消されている
* [ ] ロックファイルをコミット
* [ ] Dependabot等を有効化
* [ ] 定期セキュリティ更新を実施

## セキュリティテスト（自動化推奨）

以下の観点を自動テストに含める:

* 認証必須エンドポイントに未認証でアクセスした場合に拒否される
* 管理者権限必須エンドポイントに一般ユーザーでアクセスした場合に拒否される
* 不正入力（形式不正、型不正、制約違反）が 400 系で拒否される
* レート制限を超えたときに 429 が返る
* CSRFトークン不正時に拒否される（該当構成の場合）
* RLS/認可ポリシーにより他人データへアクセスできない（該当構成の場合）

## フレームワーク別の補足指針

### Next.js を使う場合

* 画像はフレームワークの画像最適化コンポーネントを使う
* head要素は推奨API（metadata API 等）で管理する
* 非同期データ取得は Server Components を優先し、async Client Components を避ける

### その他フレームワークでも共通する考え方

* フレームワーク標準の安全機構・最適化機構を活用する
* 危険なAPIや回避策は最小限にする
* フレームワーク規約に沿って構成し、独自ルールを増やしすぎない
